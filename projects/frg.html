frg

Possibly the worst programming language imaginable. But like in a good way.

I started this project in October 2025, and as of writing this in late March 2026, I'm still not even close to done. More info if you keep reading.

I thought having my own programming language would be cool, but it's actually very hard to make one (who knew).

Named after the 🐸 in 🐸🚀, since I love 🐸🚀.

Phase 1

"Let's make it a fast compiled language."

So I tried doing that, but it was far too complicated to make it compiled for all major platforms, especially since I also wanted to target web and have frg run in the browser.

Phase 2

"I can just make it an intepreted language."

Now I could actually do something.

First I built the lexer to seperate the raw text into tokens. This was the easy part.

Next I had to turn those tokens into an abstract syntax tree (AST). This was the hard part.

Each possible statement and expression had to be accounted for, and there were so many edge cases to figure out.

For example, you should be able to have my_function() as a standalone statement, as well as part of a larger expression.

Eventually I made so many parsing rules with so much copy pasted code, that I fell into a sort of trance and my mind started doing it automatically. I would be thinking "hmm, today I will implement the parser for dereferencing expressions" and then not having any memory of doing so.

The last step was making the actual interpreter, which was actually surprisingly easy considering the hardest part was just function execution contexts. The worst part was having an implementation that was not at all efficient, it had so much cloned date.

Eventually it was done with zero dependencies, allowing me to be able to run it on web. The problem came with the @input builtin, which would normally mean it would only be able to run in the terminal, but I had no idea how to get it to work on web, so I gave up for a while.

Phase 3

"What if it transpiled to Rust"

I honestly had no idea what was going through my head at the time. How would having Rust code that I couldn't execute in the browser help me at all.

But I implemented it anyway, and it was simple for the most part, with a few exceptions. Notably, everything had to be a mutable variable, including functions, and everything was just plopped into the main function.

Eventually, Rust lifetimes got me to give up again.

Phase 4

"What if I used Tree-sitter"

Tree-sitter is a cross-platform library used for general-purpose parsing of token streams (or something).

Basically you define a schema for your code, and Tree-sitter makes a parser that turns text into an AST. It sounds like the perfect solution, but it sucks actually.

The way you're meant to use the resulting AST is stupid, you basically have to create an object which walks through the elements, going up and down parent-children hierarchies, and constantly referring to the original source code to get actual values. Tree-sitter just tells you where the code is, now what it is.

It was still much easier to use than hardcoding everything, but having to conform to someone else's standards is basically everything I stand against.

I gave up again when I realized how hard it would be to get it to run on web.

Phase 5 (current)

"What if I replace Tree-sitter"

This is what rcket is, a generaler (more general) purpose parsing library. It's also where I'm at right now until I finish rcket.
