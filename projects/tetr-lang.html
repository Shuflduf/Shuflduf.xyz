Tetris Remakes

tetr.gd

(Godot)

The first Tetris remake I made was in Godot, originally it was meant to be made using tilemaps, but I realized that gridmaps would be cooler since they're effectively the same things but in 3D.

It was kind of ass to be honest, at one point all the code was in a single 1000+ line file, but in the end I seperated every single gameplay mechanic into it's own component.

It was originally called Trism but it didn't fit in with the naming scheme so I changed it after :)

tetr.rs.bevy

(Rust + Bevy)

I learned about Bevy and thought it was quite cool, so I thought I could use my singular previous experience to make a better version of Tetris using an ECS achitecture.

This was also the time where I was really getting into Rust, but I didn't know it nearly as well as I do now.

Yeah so it didn't go too well. The way it's structured is that each block of the piece is a seperate entity, which holds its piece index, instead of a single piece entity holding each block position.

This technically makes the system better, as you can for example remove one of the blocks and have the rest of the piece function as if it's still there, but when would anyone ever want to do that.

Also I couldn't figure out how to export it for WASM (I was still learning), so this GIF is the last gameplay that was ever played.

tetr.go

(Golang)

Started learning Go, wanted to get better at it, decided to make Tetris again (this becomes a common pattern).

This was the first Tetris clone I was actually proud of. The code is well-structured, the gameplay is smooth, and it loads in the web almost instantly.

It uses Ebitengine (it was called Ebiten when I used it for this), and honestly it was so nice to use, and exporting to web was like a single step. Honestly this project still holds up today.

tetr.rs

(Rust)

Since the disaster that was tetr.rs.bevy, I got a lot better at Rust, and I thought I would give Tetris another try, this time using Macroquad, a simpler framework that I could get to work how I wanted to.

While it was significantly better, it still kinda sucked ass. One of the main goals was to replicate the exact look and feel of tetr.go (you can tell by just looking at their demos), so I didn't do some things that would've made the code better for the sake of parity.

Still much better than using Bevy though.

tetr.rb

(Ruby)

Wanted to learn Ruby, so I made Tetris using it.

It's by far my most recent project, I only finished it yesterday.

There weren't any WASM-compatible game frameworks for Ruby, so I settled with just using ruby.wasm and an HTML canvas.

The code was messy, since it has to be partially Javascript to interact with the canvas, so I had to setup the draw loop and input handling and stuff myself. I made it easier for myself by trying less hard to replicate the exact look and feel of tetr.go

At this point I'd say I'm pretty good at making Tetris clones.

srs.json

(JSON)

Not really a Tetris clone, but it definetly helped in their creations. For tetr.gd and tetr.rs.bevy, I had all the piece information manually copied from the Tetris Wiki, but doing this was very slow and boring, so I decided to do it just once and never have to think about it again.

I'd be lying if I said it wasn't just a bunch of numbers.

A necessary evil.
